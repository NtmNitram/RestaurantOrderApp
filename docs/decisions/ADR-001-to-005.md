# ADR-001: Refresh Token en httpOnly Cookie

**Fecha**: 2026-05-15
**Estado**: Aprobado

## Contexto
JWT almacenado en localStorage es vulnerable a ataques XSS. Cualquier dependencia npm comprometida en el frontend puede leerlo y exfiltrar tokens de todos los tenants. Con una sola brecha se comprometen múltiples clientes del SaaS.

## Opciones consideradas
1. JWT en localStorage (estado actual) — simple, vulnerable a XSS
2. JWT en httpOnly cookie — protegido de XSS, requiere CSRF mitigation
3. Access token en memoria + refresh token en httpOnly cookie — mejor balance

## Decisión
**Opción 3**: access token (vida 15 min) en memoria de React (Context), refresh token (vida 7 días) en httpOnly cookie con `SameSite=Strict; Secure`. Refresh Token Rotation: cada uso genera nuevo refresh token y revoca el anterior.

## Consecuencias
- Requiere tabla `RefreshTokens` en DB
- Requiere endpoint `POST /auth/refresh`
- Requiere interceptor en frontend que maneje 401 → refresh → reintento
- Detección de robo: si refresh token "ya reemplazado" se usa → revocar cadena entera
- UX: usuario permanece logueado 7 días sin reautenticarse

# ADR-002: PWA en lugar de APK nativa

**Fecha**: 2026-05-15
**Estado**: Aprobado

## Contexto
El modelo de negocio futuro es alquilar tabletas preconfiguradas a microempresas. Se necesita decidir cómo distribuir el software.

## Opciones consideradas
1. APK Android nativa — control total, requiere firmar y distribuir APK, actualizaciones manuales
2. PWA instalable — actualización automática en todos los tenants, sin Play Store
3. Web app sin PWA — sin capacidad de instalación, UX degradada en tableta

## Decisión
**PWA instalable** con manifest.json, ícono, splash, `display: standalone`. La tableta la abre como app, el usuario no distingue de una app nativa.

## Consecuencias
- Actualización de todos los tenants en minutos con un solo deploy
- No hay fragmentación de versiones
- Offline real requiere Service Worker complejo — postergado a v2
- En v1 se requiere WiFi confiable como parte del acuerdo de onboarding
- No disponible en App Store/Play Store (no es necesario para el modelo B2B actual)

# ADR-003: Soft Delete universal en v1

**Fecha**: 2026-05-15
**Estado**: Aprobado

## Contexto
En un contexto de restaurante real, el error humano es frecuente. Un mesero puede cancelar o "borrar" algo por accidente. Sin recovery, la confianza del cliente en el sistema colapsa al primer error.

## Decisión
Todos los modelos principales (`Order`, `Client`, `MenuItem`) tienen `IsDeleted BOOLEAN DEFAULT false` y `DeletedAt TIMESTAMPTZ`. Nunca se ejecuta `DELETE` físico en v1. Los global query filters de EF Core excluyen registros con `IsDeleted = true`.

## Consecuencias
- Recovery posible para cualquier borrado accidental
- Tamaño de la DB crece con datos "borrados" (aceptable en v1, revisar en v2)
- Requiere índice parcial `WHERE IsDeleted = false` en tablas con muchos registros borrados
- La pantalla de admin debe poder listar registros borrados para recovery manual (feature mínimo en Fase 1)

# ADR-004: DeliveryRound como entidad separada (no estado en Order)

**Fecha**: 2026-05-15
**Estado**: Aprobado

## Contexto
El flujo de cierre de ronda requiere agrupar pedidos por salida del vendedor, registrar cuándo salió y cuándo regresó, y hacer conciliación de cobro + vajilla al cierre. Se necesita modelar esto en la base de datos.

## Opciones consideradas
1. Agregar campos `RoundId`, `RoundStartedAt` directamente a `Order` — simple, no escala
2. Nueva entidad `DeliveryRound` con FK en `Order` — más código, más correcto

## Decisión
**Opción 2**: entidad `DeliveryRound` (Id, RestaurantId, SellerUserId, StartedAt, ClosedAt, Status). `Order.DeliveryRoundId` FK nullable — nullable porque Mesa y Domicilio no pertenecen a una ronda.

## Consecuencias
- Reports por vendedor y por ronda son triviales con un JOIN
- Conciliación de cobro y vajilla por ronda es una query simple
- Historia de rondas disponible para análisis futuro (IA, detección de fraude)
- Solo aplica a `Client.Tipo == Externo`

# ADR-005: OrderTableware como entidad separada (no campos en OrderDetail)

**Fecha**: 2026-05-15
**Estado**: Aprobado

## Contexto
El Arca de Adán necesita rastrear piezas de vajilla (platos, vasos, cubiertos) entregadas con cada pedido externo y recuperadas al cobrar. Se necesita modelar esto sin romper `OrderDetail`.

## Opciones consideradas
1. Campos `TablewareDelivered`, `TablewareRecovered` en `Order` — no soporta múltiples tipos de vajilla
2. Campos adicionales en `OrderDetail` — rompe la semántica (OrderDetail es sobre platillos, no vajilla)
3. Nueva entidad `OrderTableware` — más correcto, extensible

## Decisión
**Opción 3**: entidad `OrderTableware` (Id, OrderId, RestaurantId, ItemType, QuantityDelivered, QuantityRecovered, DeliveredAt, RecoveredAt). Constraint DB: `QuantityRecovered <= QuantityDelivered`.

## Consecuencias
- Soporte para múltiples tipos de vajilla por pedido
- `ItemType` puede parametrizarse por tenant en v2 (tabla `TablewareTypes`)
- Solo se crea cuando `Client.Tipo == Externo`
- Al marcar Cobrado, el sistema valida que `RecoveredAt` esté poblado o que haya marcado pérdida explícita
